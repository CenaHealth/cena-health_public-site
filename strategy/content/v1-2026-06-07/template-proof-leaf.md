- **Purpose:** Per-partner outcomes-evidence proof. Where `/reference-program` is *shape-of-engagement* proof (the blueprint generalizes), the `/proof/[partner-slug]` leaf is *outcomes-evidence* proof — the partner's specific program ran, here are the artifacts, here are the outcomes, here is the partner's own attestation. The leaf does the work `/reference-program` cannot: it makes the substrate-positioning concrete in *this* partner's words, against *this* partner's numbers, with *this* partner's clearance. Per IA §4 `/proof` block + leaf template spec + cross-expert resolution #3 (`/proof` as standalone top-level spoke separates outcomes-evidence from shape-of-engagement).

Clinical champion reads clinical context + outcomes; financial steward reads program shape + outcomes + audit trail; legal/compliance reads audit trail + accountability framing in program shape; comms / external affairs reads partner-leader quote + closing testimonial (and forwards the URL as the co-announcement-safe proof artifact). The leaf must hold all four reads simultaneously without compromising any. Per IA §2 decision-role surface map + IA §4 `/proof/[partner-slug]` audience block.

- **Headline pattern (frame-setter doubles as headline):** **Partner-protagonist frame** — partner logo + partner-leader pull-quote + program name; **Cena named once in byline only.** Per IA §4 leaf template spec ("logo + leader quote + program name; Cena named once in byline") + voice.md §2 `/proof` register ("Partner-protagonist proof. ### Section 1: Partner-protagonist frame (header / hero — logo + leader quote + program name; Cena named once in byline)

**Heading:** No traditional H1. The header *is* the section: partner logo (top) + partner-leader pull-quote (center, visual weight) + program name (below quote) + single-line Cena byline (smallest visual weight). The visual hierarchy itself does the H1 work — when filled, this composition reads in two seconds: *"[Partner-leader] of [Partner-org]'s [Program name] — runs on Cena's substrate."* Anchor `#partner-frame` per IA §4 leaf template spec.

**Position:** Section 1, canonical per IA §4 leaf template spec.

**Length / shape (template):** Header block ≤ 80 words of partner-attested + Cena-byline content combined when filled. Partner logo: SVG, partner-supplied, partner-controlled rendering specifications. Partner-leader pull-quote: 1–2 sentences, partner-authored or partner-signed-off (no Cena ghostwriting). Program name: partner-attested official program name as the partner refers to it publicly. Cena byline: single line, smallest visual treatment, partner-cleared exact wording.

**What this section must do (template-shape):**

- Render the partner as the protagonist visually before any word lands. Partner logo top of viewport, partner-leader pull-quote as the visual centerpiece, program name as identifying tag, Cena byline as substrate-credit footer. The shape is the partner-protagonist frame; the words fill the shape. Per IA §4 leaf template spec + voice.md §2 `/proof` register + Phase 1 substrate-not-vendor canon.
- Specify the **partner-leader pull-quote slot** — 1–2 sentences, partner-authored or partner-signed-off, naming the program's intent in the partner's voice. The quote attribution must include: full name, title at partner-org as the partner-org publishes it, partner-org name. Per HDG row 1 partner-attested phrasing pattern + IA §4 partner-protagonist frame.
- Specify the **Cena byline slot** — single line, exact wording partner-cleared. Default template phrasing for partner clearance: *"Program runs on Cena's substrate."* Alternative partner-clearable phrasings: *"Built on Cena's clinical-intelligence substrate"* / *"Cena's substrate underlies the program."* The byline must clear partner comms before filled instance ships. Per outstanding-partner-input.md item #1 pattern.
- Specify the **partner logo slot** — partner-supplied SVG, partner-controlled rendering specifications (sizing, clear-space, monochrome-vs-color treatment, light-vs-dark variant). Partner comms approval required before filled instance ships.
- Specify the **program name slot** — partner-attested official program name as the partner refers to it publicly (matches partner's own press materials, partner site, partner publications). Verbatim, not paraphrased.

**What this section must not do (template-shape):**

- Place a Cena logo, Cena visual element, or Cena lockup at template level above the byline-credit position — collapses partner-protagonist frame. Cena visual presence on leaf URLs is byline-only. Per voice.md §2 `/proof` register + Phase 1 substrate-not-vendor canon.
- Allow partner-leader pull-quote to be Cena-authored or Cena-edited. The quote must be partner-authored or partner-signed-off verbatim. Per HDG row 4 accountability framing applied to attribution discipline + outstanding-partner-input.md item #1 partner-attestation pattern.
- Permit a partner-org logo wall, multi-partner header, or carousel framing on a single leaf URL. Each leaf is one partner, one program. Multi-partner content lives on the `/proof` collection page, not on a leaf. Per voice.md §5 forbidden patterns (no testimonials carousel; no "Trusted by [logo wall]").
- Use "case study" / "customer success" / "customer profile" anywhere in section copy or section heading — voice.md §2 `/proof` row taboo + §5 forbidden patterns.
- Cite outcome percentages, dollar figures, adherence rates, or any numeric claim in the header — outcomes ship in section 4 with primary-source citations attached. The header sets the partner-protagonist frame; section 4 substantiates with numbers. Per voice.md §5 forbidden patterns + citation spec §3 leaf row.

- Use vendor-stack names (Anthropic / Google Cloud / Vertex AI / specific EHR vendors) in the Cena byline — HDG row 3 vendor-stack non-naming applies even at byline-line scale.

**Required claims (template-shape — every filled instance carries these claim slots):**

Partner red-line clearance on exact attribution wording required before filled instance ships. Per outstanding-partner-input.md item #1 pattern applied at attribution layer.

Per outstanding-partner-input.md item #1 partner-attestation pattern.

### Section 2: Clinical context (partner's attested words about the indication / population / clinical problem)

**Heading:** Quiet H2 (writer's call at filled-instance time — e.g., *"The clinical context"* / *"What this program addresses"* / *"The indication, the population, the problem"*). Front-load load-bearing noun per voice.md §3. Anchor `#clinical-context` per IA §4 leaf template spec.

**Position:** Section 2 of IA §4 leaf template spec — establishes *why this program exists* in the partner's clinical voice before the *how it works* and *what happened* sections land.

**Length / shape (template):** 150–300 words per filled instance. Partner-attested clinical-context prose — the partner-leader, partner-clinical-team, or partner's published clinical-context material (with partner clearance) describes the indication, the population, and the clinical problem the program addresses. Cena does not narrate this section. Cena's voice appears only in section heading and any structural connective tissue (a 1-sentence Cena-side framing intro is acceptable if needed to set up the partner-attested block).

**What this section must do (template-shape):**

- Specify the **partner-attested clinical-context block** — the load-bearing slot. 150–300 words of partner-authored or partner-signed-off prose describing the indication, the population, and the clinical problem in the partner's clinical voice. Per HDG row 1 partner-attested phrasing pattern.
- Specify a **1-sentence Cena-side framing intro slot (optional)** — used only when the section needs a connective sentence to set up the partner-attested block. Default template phrasing: *"In [Partner-leader]'s words:"* / *"From [Partner-org]'s clinical team:"* Cena byline-voice, factual, sets up partner attribution without speaking *for* the partner. Per voice.md §2 `/proof` register.
- Specify the **clinical context's scope:** indication (the clinical condition or risk-state the program addresses), population (the patient cohort served — described in clinical terms, never in identity terms or demographic-targeting terms beyond what the partner's own published materials use), clinical problem (the specific clinical-care gap or outcome-improvement opportunity the program addresses).
- Hold partner-protagonist throughout the section. Cena does not characterize the indication. Cena does not characterize the population. Cena does not characterize the clinical problem. Per voice.md §2 `/proof` register + Phase 1 substrate-not-vendor canon.
- Specify that **all clinical claims about the indication / population / clinical problem are partner-attested** — every clinical statistic about the indication, every claim about the population, every clinical-problem framing comes from partner-attested source material (partner's published clinical-context material, partner-comms-cleared writing, or partner-signed-off prose drafted by Cena for partner clearance).

**What this section must not do (template-shape):**

- Allow Cena to narrate the clinical context. The clinical context is the partner's clinical voice; Cena does not have clinicians on the partner's care team during pilots (per HDG row 4 + Cena accountability model), and Cena's voice characterizing a partner's clinical-care population reads as overreach. Per HDG row 4 + voice.md §2 `/proof` register.
- Cite outcome percentages, adherence rates, or dollar figures in the clinical context — outcomes ship in section 4 with primary-source citations. Section 2 establishes *why the program exists*; section 4 substantiates *what happened*. Per citation spec §3 leaf row + voice.md §5.
- Use identity-language for the population beyond what the partner's own published materials use. Population descriptors are clinical-state (e.g., HIV-positive, diabetes-positive, post-discharge); identity-language descriptors (race, ethnicity, gender identity beyond clinical relevance) appear only when the partner's own published materials use them. Per voice.md §5 + Phase 1 dignified-representation discipline.
- Render decorative imagery (patient portraits, clinical-setting photography, food imagery) — voice.md §5 site-wide forbidden patterns.
- Name vendor-stack systems (Anthropic / Google Cloud / specific EHR vendors) — HDG row 3.
- Use "case study" / "customer" framing anywhere — voice.md §2 + §5.
- Allow Cena to make clinical-statistical claims about the indication. Even if Cena holds peer-reviewed-literature citations supporting a claim about the indication's prevalence or clinical-care gap, in this section the partner's clinical voice owns the framing. Cena-side claims about indication / population / clinical problem belong on `/for-clinical-leaders` or `/reference-program`, not on the partner's proof leaf.

**Required claims (template-shape):**

- **Partner-attested clinical-context block:** indication + population + clinical problem in the partner's clinical voice. **Citation source per filled instance:** Partner-attested per citation spec §2(c) + §9 worked example (c) pattern. Partner clearance covers every clinical claim in the block. **Required proof / artifacts:**

- **Per filled instance:** dated written record of partner clearance covering the clinical-context block (the same partner-clearance record that covers section 1 can cover section 2 — single partner-clearance file per leaf).

**Voice cues:**

- Test the partner-attested block at filled-instance time against the partner's own published clinical-context material — does the block read as something the partner would have authored independently, with Cena's job being to host it on the leaf URL? ### Section 3: Program shape (links to `/reference-program` with case-specific annotations)

**Heading:** Quiet H2 (writer's call at filled-instance time — e.g., *"How the program runs"* / *"This program's shape on the substrate"* / *"The blueprint, with this program's specifics"*). Front-load load-bearing noun per voice.md §3. Anchor `#program-shape` per IA §4 leaf template spec.

**Position:** Section 3 of IA §4 leaf template spec — bridges *why* (section 2) and *what happened* (section 4) by rendering the program's structural shape. Reuses `/reference-program`'s blueprint primitives with case-specific annotations rather than re-rendering them from scratch.

**Length / shape (template):** 200–400 words per filled instance + per-primitive annotation slots + outbound links to `/reference-program` primitives. The section does not re-render the patient-journey timeline / staffing model / data-flow architecture diagram / reporting cadence table; it links to each on `/reference-program` with case-specific annotation about how this partner's program instantiates each primitive.

**What this section must do (template-shape):**

- Specify the **primitive-by-primitive annotation pattern** — for each of the four `/reference-program` primitives (patient-journey timeline, staffing model, data-flow architecture diagram, reporting cadence table), the section renders a 40–80-word annotation describing how *this* partner's program instantiates the primitive, plus an inline link to the corresponding `/reference-program` anchor (`#patient-journey` / `#staffing-model` / `#data-flows-reporting-cadence`).
- Specify the **case-specific specifics slot per primitive:**
  - **Patient-journey timeline annotation:** name what's identical to the `/reference-program` blueprint and what's specific to this partner's program (e.g., partner-specific referral pathway, partner-specific enrollment workflow, partner-specific re-assessment cadence). Partner-attested per citation spec §2(c).
  - **Staffing model annotation:** name the partner-staffed indication-specific clinical role for this program (e.g., HIV-specialist physician for HIV-nutrition; endocrinologist for diabetes; nephrologist for CKD) and any partner-specific staffing variation (e.g., partner-specific patient-navigator structure). Per IA §7 staffing-model worked-examples pattern.
  - **Data-flow architecture annotation:** name the partner's EHR (Epic / Cerner / Athena / partner-specific — **only when partner has cleared EHR-naming for public use**; default to *"the partner's EHR system"* if not cleared) and the partner's reporting destinations. Per HDG row 3 vendor-stack non-naming (applies to partner's own EHR vendor if partner has not cleared naming) + HDG packet §3.4 row "Epic / Cerner / Athena named integrations" MUST-NOT-SAY pattern at Cena-side (partner-side EHR naming is partner's call to clear; default to generic).
  - **Reporting cadence annotation:** name the cadence-table rows that this partner's program produces and the cadence-table rows that don't apply to this program. Partner-attested per citation spec §2(c).
